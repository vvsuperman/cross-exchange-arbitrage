# OB AI Code Core Rules

## Core Rule

优先使用基于当前页面真实逻辑的最小实现；禁止预防式通用化；任何兼容性分支都必须由已复现的问题驱动。

这条规则适用于所有代码，不仅仅是 UI 自动化。

## Required Interpretation

1. 写代码时先解决当前问题，不为假设中的未来场景预埋复杂抽象。
2. 能用当前页面、当前接口、当前业务路径上的直接实现解决，就不要先写通用框架、公共基类、可插拔扩展点或多层封装。
3. 每增加一个 fallback、兼容分支、事件模拟、全局扫描或抽象层，都必须能回答：
   这个分支是在解决哪个已经复现的问题？
4. 如果说不清具体失败案例，就不应该引入这层复杂度。
5. 局部问题优先局部修复，不把单点问题扩散成全局基础设施。

## Scope

- 该规则适用于 UI 自动化、业务逻辑、交易流程、数据模型、接口封装、配置结构、脚本工具以及辅助模块。
- 禁止因为“以后可能支持更多场景”就提前做通用化设计。
- 禁止为了形式上的优雅，把单一需求改写成抽象体系。

## UI Automation Guidance

- 优先使用贴合真实 DOM 的简单选择器和直接点击。
- 只有在原生 `click()` 已被证明失效时，才增加事件派发、遍历扫描或 shadow DOM 兼容。
- 只有在同一页面已经确认存在多种稳定结构时，才允许写多分支适配。

## Review Standard

遇到自动化逻辑变复杂时，先反问三件事：

1. 这是不是当前页面的最小实现？
2. 这层兼容是不是由真实失败案例驱动？
3. 去掉这层复杂度后，当前问题是否仍然可以解决？

如果第 1 或第 2 个问题回答是否定的，就应该继续简化实现。
